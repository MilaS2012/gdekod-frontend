# База данных ГдеКод (Azure SQL)

Эта папка содержит схему и сид-данные для каталога магазинов и промокодов ГдеКод. **Персональных данных пользователей здесь нет** — только публичный каталог партнёров и сами промокоды.

## Содержимое папки

| Файл | Что делает |
|---|---|
| `schema.sql` | Создаёт все таблицы, индексы и ограничения. Запускается один раз при первом развёртывании БД (и потом ещё раз при добавлении новых таблиц). Скрипт идемпотентный — повторный запуск ничего не сломает. |
| `seed.sql` | Заполняет БД тестовыми данными: 6 категорий, 15 магазинов, 30 промокодов. Тоже идемпотентный (`MERGE`). |
| `connection-example.md` | Примеры строк подключения для Node.js, .NET и `sqlcmd`. С плейсхолдерами вместо реальных паролей. |
| `README.md` | Этот файл. |

## Как подключиться

1. Создай Azure SQL Database через Azure Portal (это будет следующий этап).
2. Возьми **server name** (вида `xxx.database.windows.net`) и **admin login**.
3. Открой Azure Portal → твой SQL Server → **Firewalls and virtual networks** → добавь свой текущий IP в whitelist (иначе подключение отвалится).
4. Подключись через любой клиент:
   - **Azure Data Studio** (бесплатный, кроссплатформенный, рекомендую)
   - **SQL Server Management Studio** (только Windows)
   - `sqlcmd` из терминала (см. `connection-example.md`)

## Как применить миграции

Порядок строго такой: сначала схема, потом сид.

### Через Azure Data Studio
1. Подключись к нужной БД.
2. Открой `schema.sql` → **Run** (F5).
3. Открой `seed.sql` → **Run**.
4. Проверь: `SELECT COUNT(*) FROM dbo.merchants;` — должно вернуться 15.

### Через `sqlcmd`
```bash
sqlcmd -S your-server.database.windows.net -d gdekod -U your-admin -P 'your-password' -i schema.sql
sqlcmd -S your-server.database.windows.net -d gdekod -U your-admin -P 'your-password' -i seed.sql
```

> **Замечание про идемпотентность:** оба скрипта можно запускать повторно. `schema.sql` использует `IF OBJECT_ID(...) IS NULL`, `seed.sql` — `MERGE`. То есть при повторном запуске данные не задвоятся, а обновятся.

## Описание таблиц

### `categories` — справочник категорий

Маленькая lookup-таблица для категорий магазинов. Берётся из фронта (вкладки «Еда», «Одежда», «Электроника» и т.д.).

| Поле | Тип | Описание |
|---|---|---|
| `id` | `INT` PK | Автоинкремент. |
| `key` | `NVARCHAR(32)` UNIQUE | Машинный ключ: `eda`, `odezhda`, `elektronika`, `travel`, `krasota`, `other`. Используется в коде на фронте и в `merchants.category`. |
| `label_ru` | `NVARCHAR(64)` | Подпись для пользователя: «Еда и продукты», «Одежда и обувь» и т.д. |
| `sort_order` | `INT` | Порядок отображения вкладок (меньше — раньше). |

### `merchants` — магазины-партнёры

Каталог магазинов: Wildberries, Ozon, Авиасейлс и т.д.

| Поле | Тип | Описание |
|---|---|---|
| `id` | `BIGINT` PK | Автоинкремент. |
| `name` | `NVARCHAR(128)` | Название магазина, как показываем пользователю («Wildberries», «М.Видео»). |
| `domain` | `NVARCHAR(255)` UNIQUE | Основной домен (`wildberries.ru`). По нему делаем поиск и сопоставление при автопроверке. **Уникален** — один домен = один магазин. |
| `category` | `NVARCHAR(32)` FK → `categories.key` | К какой категории относится магазин. |
| `logo_url` | `NVARCHAR(512)` NULL | Ссылка на логотип. Пока `NULL` для всех — заполним позже. |
| `created_at` | `DATETIME2(0)` | Когда добавили в БД. UTC. |
| `is_active` | `BIT` | `1` — показываем в каталоге. `0` — скрыли (магазин ушёл из партнёров, но удалять жалко из-за истории). |

**Индексы:** `IX_merchants_domain` — для быстрого поиска по домену.

### `coupons` — промокоды

Главная таблица. Каждая строка — один действующий или когда-то действовавший промокод.

| Поле | Тип | Описание |
|---|---|---|
| `id` | `BIGINT` PK | Автоинкремент. |
| `merchant_id` | `BIGINT` FK → `merchants.id` | К какому магазину относится. При удалении магазина — каскадное удаление промокодов (`ON DELETE CASCADE`). |
| `code` | `NVARCHAR(64)` | Сам промокод (то, что юзер копирует и вставляет на сайте магазина): `WB500RUB`, `OZON10NEW`. |
| `discount` | `NVARCHAR(64)` | Короткое отображение скидки для UI: `−500 ₽`, `−15%`. Хранится строкой намеренно — у магазинов скидки разной природы (фикс, процент, бесплатная доставка) и парсить их в число пока бессмысленно. |
| `description` | `NVARCHAR(512)` | Условия применения: «На заказ от 3 000 ₽», «Только для новых». |
| `expires_at` | `DATETIME2(0)` NULL | Когда истекает. **`NULL` = бессрочный** (пока магазин сам не отзовёт). |
| `last_checked_at` | `DATETIME2(0)` NULL | Когда автопроверка последний раз убедилась, что код жив. `NULL` = ещё не проверяли. |
| `status` | `NVARCHAR(32)` CHECK | Жизненный цикл (см. ниже). |
| `verified_text` | `NVARCHAR(64)` NULL | Человекочитаемая «свежесть» проверки для UI: `2 ч`, `5 ч`. Денормализация — храним готовую строку, чтобы не вычислять на каждый рендер. Пересчитывается раз в N минут вместе с автопроверкой. |
| `created_at` | `DATETIME2(0)` | Когда добавили промокод в БД. UTC. |

**Уникальность:** `(merchant_id, code)` — один и тот же код у одного магазина не задваивается.

**Возможные значения `status` (CHECK CONSTRAINT):**

| Статус | Когда |
|---|---|
| `active` | Промокод работает, показываем юзеру. |
| `expired` | Прошла дата `expires_at` или автопроверка вернула «не работает по причине срока». |
| `needs_manual_check` | Автопроверка не смогла однозначно определить — нужен глаз человека. |
| `removed` | Магазин убрал промокод. **Soft-delete:** строку не удаляем, чтобы не ломать историю в `coupon_checks`. |

**Индексы:**
- `IX_coupons_merchant_id` — список промокодов конкретного магазина.
- `IX_coupons_status` — фильтр «только активные» в каталоге.
- `IX_coupons_expires_at` — найти всё, что истекает скоро (для воркера-чекера).

### `coupon_checks` — история автопроверок

Append-only лог: каждая попытка автопроверки промокода — отдельная строка. Никогда не апдейтится, только вставляется. Нужен для:
- отладки («почему вы решили, что код не работает?»),
- статистики качества автопроверки,
- ручного разбора `needs_manual_check`.

| Поле | Тип | Описание |
|---|---|---|
| `id` | `BIGINT` PK | Автоинкремент. |
| `coupon_id` | `BIGINT` FK → `coupons.id` | Какой промокод проверяли. `ON DELETE CASCADE` — если удалили промокод, удаляем и его историю. |
| `checked_at` | `DATETIME2(0)` | Момент проверки (UTC). |
| `result` | `NVARCHAR(32)` CHECK | Тот же набор значений, что у `coupons.status`. Что чекер «увидел» в этот раз. |
| `raw_response_text` | `NVARCHAR(MAX)` NULL | Сырой ответ от чекера/скрейпера (HTML-фрагмент, JSON, текст ошибки). Для разбора инцидентов. |

**Индекс:** `IX_coupon_checks_coupon_id (coupon_id, checked_at DESC)` — быстро получить «последние N проверок этого промокода».

## Что НЕ в этой БД

Чтобы не было путаницы:

- **Пользователи, авторизация, сессии** — в этой схеме их нет. Для каталога+промокодов они не нужны. Когда появится профиль/избранное/история — это будет отдельная миграция (новые таблицы `users`, `user_favorites`).
- **Логи фронта, метрики, аналитика** — это не задача SQL, они идут в Application Insights / лог-агрегатор.
- **Файлы (логотипы)** — в БД храним только ссылку (`merchants.logo_url`), сами файлы лежат в Azure Blob Storage / CDN.

## Соглашения

- **Все названия — на английском** (стандарт). Описание в этом README и комментарии в `seed.sql` — на русском.
- **Время — всегда UTC** (`SYSUTCDATETIME()`). Конвертация в локальное — на фронте.
- **Стандарт T-SQL** для Azure SQL Database. На MySQL/Postgres переносить нельзя без правок (`IDENTITY`, `NVARCHAR`, `MERGE`, `OBJECT_ID` — это T-SQL-специфика).
